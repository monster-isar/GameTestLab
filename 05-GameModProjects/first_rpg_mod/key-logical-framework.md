# 核心问题解决与模块添加
## 核心问题
### 问题一：新角色入队后，原场景中角色未移除
- **表现**：新角色入队操作后成功执行入队操作，但原场景中的该角色仍存在
- **我的排查**：
  1. 搜索查找正常角色的入队控制模块；
  2. 与为新角色设置的入队控制进行对比；
  3. 通过对比，发现正常角色设置了对应的队伍状态控制参数，与队伍相关控制参数相对应。
- **我的解决**：
  调整角色参数设置，为新角色设定状态控制参数。
### 问题二：新角色无法正常离队
- **表现**：新角色可以正常入队，但离队操作无效。
- **我的排查**：
  1. 搜索查找正常角色的离队控制模块；
  2. 查询发现队伍控制使用的插件，阅读插件内容；
  3. 参考已搜索到的部分对新角色的离队控制进行调整，发现问题依旧存在；
  4. 对找到的队伍控制模块部分进行分析，发现一个重置所有正常角色状态的event，包含了正常角色的状态控制参数列表。
- **我的解决**：将新角色的状态控制参数加入发现的列表中。
- 这个方案直接解决了存在的问题，但**维护性差**，若未来再加入新角色，还需要对列表再次修改。但在当前项目中，这是最高效、稳定的选择。
### 问题三：新角色在特定切换地图时贴图丢失
- **表现**：新角色在地图上正常显示，但在切换地图后贴图不可见，交互正常。
- **我的排查**：
  1. 手动测试观察发现，此问题仅在部分地图切换后发生；
  2. **合理假设**：地图切换流程中，可能存在某个重新加载角色资源的步骤，新角色未被该步骤覆盖到；
- **我的解决**：在地图公共事件中进行逆向分析，未能定位到加载步骤。考虑到更深入地分析资源加载管线耗时较多且效率低，我选择了直接在地图加载的公共事件中主动出发一次对新角色资源的加载。
- 我的目标是“让贴图显示”，而不是“完全理解引擎”；同时我在一个已知且稳定的事件点中追加操作，没有对原有逻辑进行修改。这样操作风险可控，且更高效地完成了目标。
- 在手动测试后游戏运行流畅，无崩溃等严重问题，证明此方法是安全可行的。
## 模块添加
### 添加一：添加新角色与场景物品的交互功能
- **我的方法**：
  1. 搜索查找其他角色与场景物品进行交互的模块；
  2. 横向对比分析，结合游戏中的实际表现确定控制模块；
  3. 搜索要交互的场景物品的配置模块；
  4. 结合前述分析结果进行调整，将调整后的交互控制模块与场景物品结合，完成设置。
- 这个方案直接完成了目标，同时没有修改原有逻辑模块，风险可控且有效。
### 添加二：添加新角色与其他角色的交互功能
- **我的方法**：
  1. 搜索其他角色的交互模块，横向对比分析；
  2. 结合游戏中的实际表现，确定游戏中部分角色“第一次与某角色交互”与“非第一次与某角色交互”的控制方法；
  3. 结合前述分析结果，为新角色配置交互模块，并根据需求设定新角色与其他角色的交互控制。
- 这个方案通过对比分析的方法，确定了所需配置的模块与标识符，并遵循系统的代码模式进行增量开发，确保新新功能在风格和兼容性上与原有系统保持一致。
- 根据新角色的设置需求，针对不同的交互对象分别设置了不同内容，丰富了游戏体验。
