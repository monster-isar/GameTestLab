# 抽卡系统测试要点分析
## 内容说明
本文档内容为我在思考“对抽卡系统进行测试的时候，需要考虑什么”时想到的一些关注点。
## 要点内容
1. 功能层面
   这是测试时最基础的部分内容。要保证单抽、十连、保底机制、卡池切换等功能能够正常触发。
2. 算法层面
   我认为这是测试时最需要关注和考虑的部分，需要重点验证动态概率的准确性。现在许多游戏卡池都存在“软保底”机制，即在达到一定抽数后概率会提升，在抽取到特定等级的卡之后概率再重置。在测试过程中需要验证这个概率提升的曲线是否符合设计、触发保底后概率和保底是否准确重置。这一过程中需要大量的抽卡模拟和数据统计。
3. 数据一致性
   在我自己游玩游戏时，发现玩家前台只能看到近一段时期内的抽卡记录；而后台数据库中可能存有全部数据（如《原神》在推出“无主的命星”时，对玩家全部的抽卡记录进行统计、计算五星溢出数量，并发放对应数量的道具）。在计算保底时，程序必须从**后台全量数据库中**获取已抽次数，而不是从前台缓存中读取，避免导致保底计算错误的情况，这是很严重的bug。同时应保证前后台数据保持一致。
4. 边界情况
   测试在极端情况下（如网络波动、用户客户端闪退等）的抽卡结果是否正确记录，同时考虑在大量玩家同时抽卡时，后台服务器的处理情况。
